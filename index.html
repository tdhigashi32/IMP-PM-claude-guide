Claude Projects

Create a Project and load your PM context

A Claude Project gives you a persistent workspace so you never have to repeat your project brief. You set it up once; Claude carries it into every conversation.

How to create a Project in claude.ai
  • Go to claude.ai and click "Projects" in the left sidebar
  • Click "New Project" and give it your project name
  • Click "Project instructions" (the settings / cog icon inside the Project)
  • Copy the template below, customise it for your project, and paste it in
  • Click Save — Claude will now remember this context in every chat in this Project
Project instructions template — customise and paste this in
Click to copy · then edit the [ ] placeholders
You are my project management thinking partner for [PROJECT NAME]. Project context: - Objective: [key question the project must answer / final deliverable] - Timeline: [start date → end date, and key milestone dates] - Budget: [budget figure, or "TBC"] - Team: [names and roles] - Key stakeholders: [names and their level of influence/interest] - Scope: IN — [what is included]. OUT — [what is excluded] Frameworks I use: BCG work-planning, RACI, Influencer Matrix, MoSCoW prioritisation, Eisenhower matrix, risk log (impact × probability), Interest-Based Relational approach for conflict. In every response: - Flag scope creep risks clearly and early - Distinguish between risks (future/uncertain) and issues (present/confirmed) - Keep recommendations action-oriented and concise - When I share a draft document, suggest concrete edits rather than general comments Current project status: [e.g. "Scoping phase — kickoff not yet held"]
Use a real project you're working on right now — the more specific you are about scope and stakeholders, the more useful Claude's responses will be.
2
Artifact files

Create your five living PM documents

Each document below lives inside your Claude Project as a named Artifact. Claude generates a complete first draft from a single prompt — then you iterate and update it throughout the project. Work through them in order: the work plan comes first, everything else builds on it.

How to create a named Artifact: Paste the prompt into a new chat inside your Project. When Claude responds, click the document icon that appears in the top-right of the response to save it as a named Artifact. Give it the filename shown below so it's easy to find later.
Document 1 — work-plan.md
What it is: The living spine of your project. A table of modules, tasks, owners, due dates, and status — structured so you can see the whole project at a glance.

When to update it: After every sprint retrospective, whenever scope changes, and before each steering committee meeting.
Columns to include
Module · Task · Owner · Due date · Status · Key deliverable
Status values to use
🟢 On track · 🟡 At risk · 🔴 Off track · ✅ Done
📋 Generate prompt — paste into a new chat in your Project
Using the project context in your instructions, create a first-draft work plan as a markdown table. Columns: Module | Task | Owner | Due date | Status | Key deliverable Rules: - Break the project into 4–6 logical modules - List 2–5 tasks per module - Use placeholder names where I haven't specified owners yet - Set all statuses to "🟢 On track" for now - Flag any tasks where the scope is ambiguous — add a note in the Key deliverable column Save this as an Artifact called work-plan.md
🔄 Update prompt — use this after each sprint or scope change
Open the work-plan.md artifact. Here are the updates from this week: Completed tasks: [list] Tasks now at risk: [list + reason] New tasks to add: [list + module they belong to] Scope changes: [describe any changes] Update the table accordingly and highlight what changed.
After Claude generates the first draft, ask: "Now challenge your own output — what are the three weakest assumptions in this work plan?" This surfaces gaps before your kickoff.
Document 2 — raci-matrix.md
What it is: A grid showing every stakeholder's role (Responsible / Accountable / Consulted / Informed) across every key activity. Prevents confusion about who owns what and who needs to be kept in the loop.

When to update it: After your kickoff meeting, whenever the team changes, and when new activities are added to the work plan.
R
Does the work
A
Final decision owner
C
Must be asked before decision
I
Told after decision
📋 Generate prompt
Create a RACI matrix for my project as a markdown table. Stakeholders (rows): [list names and roles — e.g. "Sam, Project Manager · Javier, Senior Manager · Helen, Team Lead · Margarita, Vendor"] Activities (columns): [list 6–10 key project activities from the work plan] For each cell, assign R, A, C, or I. Then below the table: - Flag any activity with no Accountable owner - Flag any activity where more than two people are Responsible (usually means unclear ownership) - Flag any stakeholder who appears nowhere (possible gap) Save this as an Artifact called raci-matrix.md
🔍 Sense-check prompt — run after your kickoff meeting
Open the raci-matrix.md artifact. In our kickoff meeting, the following things changed or were clarified: [describe any role changes, new stakeholders, or activities that were added/removed] Update the matrix and re-run the gap check.
Document 3 — risk-log.md
What it is: A live register of every identified risk — what it is, how likely it is, how bad it would be, and what you're doing about it. The key discipline is distinguishing risks (future, uncertain) from issues (present, happening now).

When to update it: Weekly as part of your project rhythm. Always before a steering committee meeting.
Risk (manage proactively)
Something that might go wrong. You have time to prevent or reduce it. Managed with a mitigation plan.
Issue (act now)
Something that has gone wrong. Needs an immediate action plan. Track separately from risks.
📋 Generate prompt
Create a risk log for my project as a markdown table. Columns: Risk ID | Risk description | Category | Probability (H/M/L) | Impact (H/M/L) | Severity score | Mitigation approach | Owner | Status Brainstorm at least 8 realistic risks for this type of project. For each: - Assign a category: Schedule / Budget / Resource / Scope / Stakeholder / External - Give a severity score: multiply probability (H=3, M=2, L=1) × impact (H=3, M=2, L=1) - Suggest a mitigation approach using one of: Eliminate / Mitigate / Transfer / Accept / Escalate Sort the table by severity score, highest first. Below the table, add a separate section called "Current Issues" for any risks that have already materialised. Save this as an Artifact called risk-log.md
🔄 Weekly update prompt
Open the risk-log.md artifact. Here is this week's risk update: New risks to add: [describe] Risks that have changed: [describe what changed and why] Risks that have materialised into issues: [describe — move these to the Issues section] Risks that can be closed: [describe] Update the log, re-sort by severity, and flag any risks that now need SteerCo attention.
Document 4 — comms-plan.md
What it is: A structured plan for getting the right message to the right stakeholder at the right time in the right way. Built on the Who / What / How framework. Prevents stakeholders from feeling out of the loop — one of the most common sources of project friction.

When to update it: When new stakeholders are added, when the project phase changes, or when someone flags they feel uninformed.
📋 Generate prompt
Create a stakeholder communication plan for my project as a markdown table. Use the Who / What / How framework. Columns: Stakeholder / Group | Role on project | What they need to know | Key message | Channel | Frequency | Owner Include a row for each stakeholder group in the project. For each row: - Tailor the "What they need to know" to their actual interests and concerns - Suggest the most appropriate channel (e.g. weekly email, 1:1 call, SteerCo deck, Slack update, monthly report) - Flag any stakeholders who are high-importance but currently low-information Below the table, add 3 tips for communicating with senior stakeholders on this project specifically. Save this as an Artifact called comms-plan.md
✉️ Draft a specific stakeholder message
Using the comms-plan.md artifact as context, draft a [weekly update / meeting invite / escalation note] to [stakeholder name / group]. Key points to include: [list 2–4 points] Tone: [e.g. reassuring / direct / collaborative] Length: [e.g. 150 words / one paragraph / one page]
Document 5 — sprint-retro-log.md
What it is: A running record of what happened in each sprint or project phase — what went well, what didn't, and what you're changing. Over time this becomes your personal PM learning record and feeds directly into your development plan.

When to update it: After every sprint retrospective (every 2–4 weeks), and after any significant project event or decision.
📋 Create the log — run this once at project start
Create a sprint retrospective log for my project. Set it up as a structured markdown document with: 1. A header section with project name, start date, and total number of sprints planned 2. A template block for each sprint entry with these fields: - Sprint number and date - Goal for this sprint - What we completed - What went well (bullet points) - What didn't go well (bullet points) - What we're changing for next sprint - Risks or scope changes identified - Team morale (1–5 scale with one-sentence note) Leave Sprint 1 blank ready to fill in after the first retrospective. Save this as an Artifact called sprint-retro-log.md
🔄 Fill in after each sprint retrospective
Open the sprint-retro-log.md artifact and fill in Sprint [N]. Here is what happened: - Goal for this sprint: [describe] - Completed: [list] - Went well: [list] - Didn't go well: [list] - Changes for next sprint: [list] - Any new risks or scope changes: [describe] - Team morale: [1–5] — [one sentence note] After filling in the entry, identify one pattern or learning that is showing up across multiple sprints.
At the end of the project, ask Claude: "Review the full sprint-retro-log.md and write a one-page summary of my biggest PM strengths and development areas from this project." This is ready-made content for your development plan.
Suggested order: Start with the work plan (Doc 1), then build the RACI once you've confirmed your team (Doc 2), then the risk log (Doc 3). The comms plan (Doc 4) can wait until after your kickoff. The retro log (Doc 5) is set up once and updated as you go.
3
Scoping

Define the key question and scope boundaries

Scoping is an alignment process, not just a planning task. Use these prompts to sharpen your key question, define what's in and out, and test your hypothesis before you start executing.

Scoping prompts
1 · Sharpen the key question
Based on the project context above, what single key question should this project answer? Draft three candidate formulations, then recommend the sharpest one with your reasoning.
2 · Check scope boundaries
Review the scope I've defined. List five items that are clearly IN scope, five that are clearly OUT, and flag any ambiguous areas that could become scope creep later. For each ambiguous item, suggest how I should clarify it with my sponsor.
3 · Build a testable hypothesis
Using a hypothesis-based approach, help me define this project in four parts: 1. Restate the key question clearly 2. Write a testable hypothesis (what we believe to be true) 3. Describe the tests or analyses that would validate or refute it 4. Define the success criterion (how we'll know we're right) Keep each part to 2–3 sentences.
Handling scope creep — the most common PM challenge
When a stakeholder asks for something new, use this prompt before your next meeting with them:
Scope creep response prompt
My sponsor [name] has just asked for [describe the new request]. This was not in the original scope. Help me draft a response that: 1. Acknowledges their request positively 2. Explains the specific impact on timeline / budget / other deliverables 3. Offers two concrete options for how we could accommodate it 4. Asks for their decision Keep it to 150 words maximum. Tone: collaborative, not defensive.
Scope change vs scope creep: Scope change is controlled and planned — you agree on it with your sponsor and adjust the plan. Scope creep is uncontrolled — you say yes to something small without assessing the knock-on effect. The prompt above turns potential scope creep into a managed scope change decision.
4
Planning & execution

Plan, prioritise, and run your sprints

Use these prompts throughout the project execution phase — to prioritise your deliverables, run sprint retrospectives, and write steering committee updates.

Phase prompts
MoSCoW prioritisation
Take the deliverables list below and categorise each as Must have / Should have / Could have / Won't have, with a one-sentence justification for each decision. Flag any where you're unsure and ask me to clarify. [paste your deliverables list here]
Eisenhower matrix for your task list
Here is my current task list: [list your tasks]. Place each task into the Eisenhower matrix quadrants (Urgent+Important / Not Urgent+Important / Urgent+Not Important / Not Important). Then give me a prioritised "do today" list of no more than five items. [paste task list here]
Sprint retrospective
We just completed sprint [N]. Here's what happened: [brief summary of work done and outcomes]. Run a quick retrospective: 1. What went well? 2. What didn't go well? 3. What should we adjust for the next sprint? 4. Are there any emerging risks or scope changes we need to flag? Then suggest the top three priorities for sprint [N+1].
After Claude gives you the retrospective, paste it into your sprint-retro-log.md artifact. Over time this becomes an invaluable record for your own development plan.
Steering committee status update
Draft a concise steering committee status update for week [N]. Format: 1. Overall RAG status (Red / Amber / Green) with a one-sentence justification 2. Progress this week — key activities completed 3. Risks in focus — top 2–3 with current mitigation status 4. Decisions needed from the SteerCo Keep to 300 words maximum. Tone: factual and direct.
Brainstorm risks
Brainstorm the top 10 risks for this project. For each risk: 1. Describe it in one sentence 2. Rate probability: High / Medium / Low 3. Rate impact: High / Medium / Low 4. Suggest a mitigation approach: Eliminate / Mitigate / Transfer / Accept / Escalate Output as a markdown table.
Distinguish risks from issues
Here is my current list of project concerns: [list]. Separate these into RISKS (future / uncertain) and ISSUES (present / already happening). For issues, suggest an immediate action plan. For risks, suggest a monitoring trigger that would tell me the risk is becoming an issue. [paste concern list here]

Stakeholders & conflict

Know your stakeholders and handle difficult conversations

Project success depends on communication — getting the right message to the right person at the right time. These prompts help you map influence, plan communication, and prepare for conflict.

Stakeholder mapping & communication
Build a RACI matrix
Create a RACI matrix for the activities below. Stakeholders: [list stakeholder names / roles] Activities: [list key project activities or workstreams] Output as a markdown table. Then flag: - Any activities with no clear Accountable owner - Any activities where too many people are listed as Responsible - Any stakeholders with no role at all (potential oversight)
Map stakeholders on the BCG influencer matrix
Using the BCG influencer matrix (2×2 grid: importance to project success vs current level of support), place each stakeholder into one of four quadrants: - Change Agents: high importance, high support → actively champion - Change Facilitators: low importance, high support → light touch updates - Major Skeptics: high importance, low support → dedicate energy here - Minor Skeptics: low importance, low support → manage negative publicity For each person, suggest one specific engagement action. Stakeholders: [list names and what you know about them]
Draft a communication plan
Build a stakeholder communication plan using the Who / What / How framework. For each stakeholder group: what do they need to know, what is the key message, what channel will you use, and how often? Output as a markdown table with columns: Stakeholder | What they need to know | Key message | Channel | Frequency
Preparing for difficult conversations
Use the Interest-Based Relational approach — focus on the relationship first, separate people from problems, then explore options together.
Prepare for a difficult conversation
I need to have a difficult conversation with [stakeholder name / role] about [topic / issue]. Help me prepare using the Interest-Based Relational approach: 1. How do I open by focusing on the relationship, not the problem? 2. How do I frame the issue without making it personal? 3. What are their likely underlying interests and concerns? 4. What questions should I ask before presenting my view? 5. What are two possible outcomes we could both accept?
Use Claude to prepare, not to replace the conversation. Run the prompt before a difficult meeting — Claude helps you think clearly about the other person's perspective and your own. The actual conversation is yours to have.

Interactive artifacts

Build interactive PM tools inside Claude

Claude can build working apps — risk heat maps, stakeholder mappers, prioritisation boards — from a single prompt. These live inside Claude Artifacts and need no technical skill to create.

Three interactive tools you can build today
HTML app
Risk heat map
Enter risks, rate probability and impact, and see them plotted on a live 3×3 heat map with colour-coded severity. Updates in real time as you add risks.
HTML app
Stakeholder influence mapper
An interactive version of the BCG influencer matrix. Add stakeholder names and place them in quadrants. Shows engagement strategy per quadrant.
React app
MoSCoW prioritisation board
Four-column board (Must / Should / Could / Won't). Add deliverables, drag between columns, see your prioritised list update live.
Build the risk heat map — paste this prompt
Click to copy · paste into a new chat in your Project
Build me an interactive risk log as an HTML artifact. Features needed: - A form to enter: risk description, probability (High / Medium / Low), impact (High / Medium / Low), mitigation approach (Eliminate / Mitigate / Transfer / Accept / Escalate) - A sortable table showing all risks entered - A 3×3 heat map grid showing risks plotted by probability and impact, colour-coded green / amber / red by severity level - All updates should happen live without reloading Make it clean and easy to use on a laptop screen.
Build the stakeholder mapper
Build an interactive stakeholder influence mapper as an HTML artifact. It should show a 2×2 BCG influencer matrix with axes: "Importance to project success" (vertical) and "Level of support" (horizontal). The four quadrants should be labelled: Change Agents (high/high), Change Facilitators (low/high), Major Skeptics (high/low), Minor Skeptics (low/low). Let me add stakeholder names and click to place them in a quadrant. Show the recommended engagement strategy for that quadrant below the matrix when a stakeholder is selected.
Power user habit: After you build an artifact, ask Claude: "Now add [feature]" or "Make the table exportable as CSV." Claude iterates on its own artifacts — you don't need to start over.

Power-user habits to build into your workflow
  • End every important chat with: "What did I miss?"
  • Upload your work plan and ask: "Spot the gaps and risks in this plan."
  • Before a difficult meeting: "Steelman my stakeholder's position — what's the strongest version of their argument?"
  • Paste a stakeholder email and ask: "What are they really asking for, and what are their underlying concerns?"
  • Paste meeting notes and ask: "Extract all action items, owners, and any risks or scope changes mentioned."
  • Ask Claude to draft the SteerCo update first — then edit it yourself rather than writing from scratch.
