1
Claude Projects

Set up a Project and load your PM context

A Claude Project is a persistent workspace — you define your project once and Claude carries that context into every conversation. This is the single highest-leverage thing you can do.

How to create a Project in claude.ai
  • Go to claude.ai and click "Projects" in the left sidebar
  • Click "New Project" and give it your project name
  • Open "Project instructions" (the cog icon inside the Project)
  • Copy the template below, fill in your project details, and paste it in
  • Click Save — Claude now remembers this in every chat inside this Project
Project instructions template — customise the [ ] placeholders, then paste in
Click to copy
You are my project management thinking partner for [PROJECT NAME]. Project context: - Objective: [the key question this project must answer, or the final deliverable] - Timeline: [start date → end date, plus key milestone dates] - Budget: [budget figure, or "TBC"] - Team: [names and roles] - Key stakeholders: [names and their level of influence / interest] - Scope: IN — [what is included]. OUT — [what is excluded] Frameworks I use: BCG work-planning, RACI, Influencer Matrix, MoSCoW prioritisation, Eisenhower matrix, risk log (impact × probability), Interest-Based Relational approach for conflict. In every response: - Flag scope creep risks clearly and early - Distinguish between risks (future/uncertain) and issues (present/confirmed) - Keep recommendations action-oriented and concise - When I share a draft document, suggest concrete edits rather than general comments Current project status: [e.g. "Scoping phase — kickoff not yet held"]
Use a real project you're currently working on. The more specific your context, the more useful every response will be.
2
Scoping

Scope your project

Scoping is an alignment process, not just a planning task. Before you build any documents, you need a sharp key question, clear in/out boundaries, and a testable hypothesis. These three prompts do that work.

Three scoping prompts — run these before building any documents
1 · Sharpen the key question
Based on the project context in your instructions, what single key question should this project answer? Draft three candidate formulations, then recommend the sharpest one with your reasoning.
2 · Set scope boundaries
Review the scope I've defined. List five items clearly IN scope, five clearly OUT, and flag any ambiguous areas that could become scope creep. For each ambiguity, suggest how to clarify it with my sponsor before kickoff.
3 · Write a testable hypothesis
Using a hypothesis-based approach, help me define this project in four parts: 1. Restate the key question clearly 2. Write a testable hypothesis (what we believe to be true) 3. Describe the analyses that would validate or refute it 4. Define the success criterion — how we'll know we're right Keep each part to 2–3 sentences.
When scope changes during the project — handling scope creep
Scope creep happens when you say yes to changes without assessing the knock-on effect. Use this prompt before responding to any new stakeholder request.
Scope creep response — use before your next meeting
My sponsor [name] has asked for [describe the new request]. This was not in the original scope. Help me draft a response that: 1. Acknowledges their request positively 2. Explains the specific impact on timeline / budget / other deliverables 3. Offers two concrete options for accommodating it 4. Asks for their decision Keep to 150 words. Tone: collaborative, not defensive.
Scope change vs scope creep: Scope change is controlled — agreed with your sponsor, with the plan adjusted accordingly. Scope creep is uncontrolled — a small yes that quietly expands the project without budget or time adjustment. This prompt converts potential scope creep into a managed decision.
3
Work plan

Build your work plan

Now that your scope is defined, build the living spine of your project. The work plan is a document you'll update throughout — after every sprint, every scope change, and every SteerCo meeting.

work-plan.md — generate your first draft
Columns to include
Module · Task · Owner · Due date · Status · Key deliverable
Status values to use
🟢 On track  ·  🟡 At risk  ·  🔴 Off track  ·  ✅ Done
📋 Generate — paste into a new chat inside your Project
Using the project context in your instructions, create a first-draft work plan as a markdown table. Columns: Module | Task | Owner | Due date | Status | Key deliverable Rules: - Break the project into 4–6 logical modules - List 2–5 tasks per module - Use placeholder names where owners are not yet confirmed - Set all statuses to "🟢 On track" for now - Flag any tasks where scope is ambiguous with a note in the Key deliverable column Save this as an Artifact called work-plan.md
🔄 Update — use after every sprint or scope change
Open the work-plan.md artifact. Here are this week's updates: Completed tasks: [list] Tasks now at risk: [list + reason] New tasks to add: [list + which module they belong to] Scope changes to reflect: [describe any] Update the table and highlight what changed.
After the first draft is generated, ask: "Challenge your own output — what are the three weakest assumptions in this work plan?" This surfaces gaps before your kickoff meeting.
Prioritising your deliverables within the work plan
Use MoSCoW when you need to cut or rank a list of deliverables and justify each decision to your sponsor.
MoSCoW prioritisation
Take the deliverables list below and categorise each as Must have / Should have / Could have / Won't have. Give a one-sentence justification for each decision and flag any where you're unsure. [paste your deliverables list here]
Use the Eisenhower matrix for your personal task list when you feel overwhelmed and need to decide what to do today.
Eisenhower matrix — daily prioritisation
Here is my current task list: [paste tasks]. Place each into the Eisenhower matrix (Urgent+Important / Not Urgent+Important / Urgent+Not Important / Not Urgent+Not Important). Then give me a "do today" list of no more than five items. [paste task list here]
4
Stakeholders

Map your stakeholders, build your RACI and comms plan

With your work plan in place, you know your activities. Now map who is responsible for each one, how you'll communicate with each stakeholder, and how to handle difficult conversations when they arise.

raci-matrix.md — who owns what
The RACI matrix maps every stakeholder against every key activity. It prevents the most common project failure mode: unclear ownership.
R
Does the work
A
Final decision owner
C
Must be asked before decision
I
Told after decision
📋 Generate — use after your work plan is drafted
Create a RACI matrix for my project as a markdown table. Stakeholders: [list names and roles — e.g. "Sam, Project Manager · Javier, Senior Manager · Helen, Team Lead · Margarita, Vendor"] Activities: [copy the 6–10 key activities from work-plan.md] For each cell assign R, A, C, or I. Then below the table flag: - Any activity with no Accountable owner - Any activity where more than two people are listed as Responsible - Any stakeholder who appears in no cell at all Save this as an Artifact called raci-matrix.md
🔍 Sense-check — run after your kickoff meeting
Open raci-matrix.md. In our kickoff the following changed: [describe any role changes, new stakeholders, or activities added/removed] Update the matrix and re-run the gap check.
comms-plan.md — the right message to the right person
Before writing the comms plan, map where each stakeholder sits on the BCG influencer matrix — this tells you how much energy to invest in each person and what kind of message they need.
Step 1 · Map stakeholders on the BCG influencer matrix
Using the BCG influencer matrix (2×2: importance to project success vs current level of support), place each stakeholder into one of four quadrants: - Change Agents: high importance, high support → help them actively champion the project - Change Facilitators: low importance, high support → light-touch updates to maintain buy-in - Major Skeptics: high importance, low support → dedicate energy here, pre-sell solutions - Minor Skeptics: low importance, low support → give a forum to voice concerns, don't overinvest For each person suggest one specific engagement action. Stakeholders: [list names and what you know about their current stance]
Step 2 · Build the comms plan using Who / What / How
Using the influencer matrix output and the project context, build a stakeholder communication plan as a markdown table. Columns: Stakeholder / Group | What they need to know | Key message | Channel | Frequency | Owner For each row tailor the message to their interests and concerns, and flag any high-importance stakeholders who are currently low-information. Below the table add 3 specific tips for managing senior stakeholders on this project. Save this as an Artifact called comms-plan.md
✉️ Draft a specific message to a stakeholder
Using comms-plan.md as context, draft a [weekly update / meeting invite / escalation note] to [stakeholder name]. Key points to cover: [list 2–4 points] Tone: [e.g. reassuring / direct / collaborative] Length: [e.g. 150 words / one paragraph]
Preparing for difficult conversations
Use the Interest-Based Relational approach — focus on the relationship first, separate people from problems, then explore options together. Run this prompt before the meeting, not in it.
Difficult conversation prep
I need to have a difficult conversation with [stakeholder name / role] about [topic]. Help me prepare using the Interest-Based Relational approach: 1. How do I open by focusing on the relationship, not the problem? 2. How do I frame the issue without making it personal? 3. What are their likely underlying interests and concerns? 4. What questions should I ask before presenting my view? 5. What are two outcomes we could both accept?
Claude prepares you — it doesn't replace the conversation. Use the output to think clearly about the other person's perspective before you walk into the room.
5
Risk management

Manage risk

Every project has risks. The discipline is identifying them early, scoring them honestly, and knowing the difference between a risk you can still prevent and an issue you must act on now.

risk-log.md — your live risk register
Risk → manage proactively
Something that might go wrong. You still have time to prevent or reduce it. Managed with a mitigation plan and a monitoring trigger.
Issue → act now
Something that has gone wrong. Needs an immediate action plan. Tracked separately in the Issues section of the log.
📋 Generate — run once after your work plan is in place
Create a risk log for my project as a markdown table. Columns: Risk ID | Risk description | Category | Probability (H/M/L) | Impact (H/M/L) | Severity score | Mitigation approach | Owner | Status Brainstorm at least 8 realistic risks. For each: - Category: Schedule / Budget / Resource / Scope / Stakeholder / External - Severity score: probability (H=3 M=2 L=1) × impact (H=3 M=2 L=1) - Mitigation: Eliminate / Mitigate / Transfer / Accept / Escalate Sort by severity score, highest first. Below the table add a separate "Current Issues" section for any risks that have already materialised. Save this as an Artifact called risk-log.md
🔄 Weekly update — run before every SteerCo
Open risk-log.md. Here is this week's update: New risks to add: [describe] Risks that have changed: [what changed and why] Risks that have become issues: [describe — move to Issues section] Risks that can be closed: [describe] Update the log, re-sort by severity, and flag any risks that now need SteerCo attention.
🔍 Classify a list of concerns — risks vs issues
Here is my current list of project concerns: [list]. Separate these into RISKS (future/uncertain) and ISSUES (present/already happening). For issues suggest an immediate action plan. For risks suggest a monitoring trigger — the signal that would tell me the risk is becoming an issue. [paste concern list here]

Execution & control

Run your sprints and control the project

With all your documents in place, this step covers the ongoing rhythm of project execution — sprint retrospectives, steering committee updates, and the sprint retro log that tracks your learning over time.

Execution prompts
Run a sprint retrospective
We just completed sprint [N]. Here's what happened: [brief summary of work and outcomes]. Run a retrospective: 1. What went well? 2. What didn't go well? 3. What should we change for the next sprint? 4. Any emerging risks or scope changes to flag? Then suggest the top three priorities for sprint [N+1].
Draft a SteerCo status update
Draft a steering committee status update for week [N]. Format: 1. Overall RAG status (Red / Amber / Green) with a one-sentence justification 2. Progress this week — key activities completed 3. Risks in focus — top 2–3 from risk-log.md with current mitigation status 4. Decisions needed from the SteerCo Keep to 300 words. Tone: factual and direct.
sprint-retro-log.md — your project learning record
The retro log captures what happened in each sprint over the full project lifecycle. Over time it becomes your personal PM development record — and feeds directly into your post-project development plan.
📋 Create the log — run this once at project start
Create a sprint retrospective log for my project as a structured markdown document. Include: 1. A header section: project name, start date, sprints planned 2. A reusable entry template for each sprint with fields: Sprint number + date · Goal · What we completed · What went well · What didn't go well · What we're changing next sprint · Risks or scope changes identified · Team morale (1–5 + one sentence) Leave Sprint 1 blank, ready to fill in after the first retrospective. Save this as an Artifact called sprint-retro-log.md
🔄 Fill in after each sprint retrospective
Open sprint-retro-log.md and fill in Sprint [N]: Goal: [describe] Completed: [list] Went well: [list] Didn't go well: [list] Changing next sprint: [list] New risks or scope changes: [describe] Team morale: [1–5] — [one sentence] After filling in the entry, identify one pattern showing up across multiple sprints.
At project end, ask: "Review the full sprint-retro-log.md and write a one-page summary of my biggest PM strengths and development areas." This is ready-made content for your development plan.

Interactive artifacts

Build interactive PM tools inside Claude

Claude can build working apps from a single prompt — no technical skill required. These three tools bring your PM documents to life as interactive, visual dashboards.

Three interactive tools — one prompt each
An interactive app where you enter risks and see them plotted live on a colour-coded probability/impact heat map.
Build the risk heat map
Build an interactive risk log as an HTML artifact. Features: - A form to enter: risk description, probability (H/M/L), impact (H/M/L), mitigation approach (Eliminate / Mitigate / Transfer / Accept / Escalate) - A sortable table of all risks entered - A 3×3 heat map grid with risks plotted by probability and impact, colour-coded green / amber / red All updates should happen live without reloading. Clean layout for a laptop screen.
An interactive BCG influencer matrix. Add stakeholder names, place them in quadrants, and see the recommended engagement strategy for each.
Build the stakeholder influence mapper
Build an interactive stakeholder influence mapper as an HTML artifact. Features: - A 2×2 BCG influencer matrix with axes: "Importance to project success" (vertical) and "Level of support" (horizontal) - Four labelled quadrants: Change Agents (high/high) · Change Facilitators (low/high) · Major Skeptics (high/low) · Minor Skeptics (low/low) - Ability to add stakeholder names and click to place them in a quadrant - When a stakeholder is selected, show the recommended engagement strategy for their quadrant
A four-column drag-and-drop board for categorising your deliverables. Add items, move them between buckets, and export the result.
Build the MoSCoW prioritisation board
Build a MoSCoW prioritisation board as an HTML artifact. Features: - Four columns: Must have / Should have / Could have / Won't have - Cards that can be moved between columns by clicking - Each card shows the deliverable name and a one-line justification - A text input to add new items - A count of items in each column displayed in the column header
Iterate on any artifact. After Claude builds a tool, just say "Add [feature]" or "Make the table exportable as CSV." Claude edits its own output — you never need to start over.

Power-user habits to build into your workflow
  • End every important Claude chat with: "What did I miss?"
  • Upload your work plan and ask: "Spot the gaps and risks in this plan."
  • Before a difficult meeting: "Steelman my stakeholder's position — what's the strongest version of their argument?"
  • Paste a stakeholder email and ask: "What are they really asking for, and what are their underlying concerns?"
  • Paste meeting notes and ask: "Extract all action items, owners, and any risks or scope changes mentioned."
  • Draft the SteerCo update with Claude first — then edit it yourself rather than writing from scratch.
